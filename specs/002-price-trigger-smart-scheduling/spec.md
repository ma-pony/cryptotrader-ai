# 功能规格说明书：价格触发器与智能调度

**Feature Branch**: `002-price-trigger-smart-scheduling`
**Created**: 2026-04-17
**Status**: Draft

---

## 背景与动机

CryptoTrader-AI 当前的调度系统（`scheduler.py`）仅支持两种触发模式：

- **Interval 触发**：固定时间间隔（`interval_minutes`，默认 240 分钟）
- **Cron 触发**：每日固定时刻（`daily_summary_hour`，默认 UTC 0 点）

这意味着系统在价格剧烈波动时无法即时响应，只能等待下一个固定周期。在高波动市场中，这会导致错过最佳入场时机，或未能及时规避风险。

本功能为调度系统引入**价格触发器**（基于 Binance WebSocket 实时价格流）、**Telegram bot 通知**、**前端调度管理页**，以及**Agent 自我调度**能力。

---

## 用户场景与验收标准

### 用户故事 1 — 价格突破触发分析 (Priority: P0)

**场景描述**：用户希望当 BTC/USDT 突破某个价格阈值时（例如跌破 80,000 USDT），系统自动触发一次完整的交易分析流程，而不必等待下一个 4 小时周期。

**优先级说明**：这是本功能最核心的价值主张——把调度从"时间驱动"升级为"事件驱动"，直接影响交易时效性。

**独立验收测试**：关闭 interval 调度，仅配置一条绝对阈值价格触发规则，观察价格穿越阈值后是否在 5 秒内触发分析，且不重复触发（冷却期内）。

**验收场景**：

1. **Given** 用户配置了一条规则"BTC/USDT 价格低于 80,000 USDT 时触发"，**When** Binance WebSocket 推送的价格首次低于 80,000 USDT，**Then** 系统在 5 秒内触发 BTC/USDT 的完整分析周期，并在触发日志中记录触发原因。

2. **Given** 上述规则已触发，**When** 价格在冷却期（默认 30 分钟）内持续低于阈值，**Then** 系统不重复触发，仅记录"冷却中"状态。

3. **Given** 价格触发器已启用，**When** WebSocket 连接断开超过 30 秒，**Then** 系统自动尝试重连，并在通知系统中发出 `ws_reconnect` 事件，不丢失已配置的规则。

---

### 用户故事 2 — 百分比变化触发 (Priority: P0)

**场景描述**：用户希望当某交易对价格在过去 N 分钟内涨跌幅超过 X% 时触发分析，以捕捉异常波动信号。

**优先级说明**：百分比变化比绝对阈值更通用，适用于多种币对而无需手动维护价格水位。

**独立验收测试**：配置"ETH/USDT 15 分钟内跌幅超过 3% 时触发"，使用历史价格回放验证触发时机准确。

**验收场景**：

1. **Given** 规则"ETH/USDT 15 分钟内价格变化幅度超过 ±3%"，**When** 过去 15 分钟的收盘价相比当前价变化超过 3%，**Then** 系统触发 ETH/USDT 分析并在触发记录中记录变化幅度（如 `-3.7%`）。

2. **Given** 多个交易对都配置了百分比触发规则，**When** 同一时刻多个交易对同时满足条件，**Then** 系统并发触发所有满足条件的交易对分析，且各对互不阻塞。

---

### 用户故事 3 — 资金费率异常触发 (Priority: P1)

**场景描述**：用户希望当某交易对的资金费率超过异常阈值时（例如超过 0.1%），系统自动触发分析，以利用资金费率套利或提前规避极端市场。

**优先级说明**：资金费率异常是加密市场特有的高价值信号，现有系统已在 `VolatilityConfig.funding_rate_threshold` 中有相关阈值，但仅用于风控拦截，未能驱动主动分析。

**独立验收测试**：配置资金费率触发规则，通过 mock Binance API 注入超阈值资金费率数据，验证触发行为。

**验收场景**：

1. **Given** 规则"BTC/USDT 资金费率绝对值超过 0.1% 时触发"，**When** 轮询获取的资金费率为 `+0.15%`，**Then** 系统触发 BTC/USDT 分析，并在触发原因中标注 `funding_rate=+0.15%`。

2. **Given** 资金费率触发已激活，**When** 连续三次轮询资金费率都超阈值，**Then** 冷却期机制确保仅在第一次触发，后续在冷却期内不重复触发。

---

### 用户故事 4 — 连续 K 线模式触发 (Priority: P1)

**场景描述**：用户希望当价格出现特定 K 线序列（例如"连续 3 根 1 小时阴线"）时触发分析，以捕捉趋势转变信号。

**优先级说明**：K 线模式比单点价格更能反映市场结构变化，但实现复杂度较高，因此优先级低于绝对阈值和百分比变化。

**独立验收测试**：配置"连续 3 根 1 小时阴线后触发"规则，使用历史 OHLCV 数据回放验证模式识别准确率。

**验收场景**：

1. **Given** 规则"BTC/USDT 1h 连续 3 根阴线（close < open）"，**When** 最近三根已收盘的 1h K 线均为阴线，**Then** 系统触发 BTC/USDT 分析，触发原因中包含识别到的 K 线模式描述。

2. **Given** 上述规则，**When** 只有 2 根连续阴线（第 3 根为阳线），**Then** 系统不触发，规则状态保持"等待中"。

---

### 用户故事 5 — Telegram 通知触发事件 (Priority: P1)

**场景描述**：用户希望当价格触发器激活时，能够通过 Telegram bot 及时收到通知，包括触发原因、触发时间和当前价格。

**优先级说明**：现有通知系统（`notifications.py`）只支持通用 webhook，不支持 Telegram 原生格式。Telegram 是交易者最常用的实时通知渠道。

**独立验收测试**：配置 Telegram Bot Token 和 Chat ID，触发一条价格规则，验证 Telegram 消息在 10 秒内到达，格式正确。

**验收场景**：

1. **Given** 已配置 Telegram Bot Token 和 Chat ID，且通知事件列表包含 `price_trigger`，**When** 价格触发器激活，**Then** Telegram 频道在 10 秒内收到消息，消息包含：交易对、触发类型、触发价格、触发时间（UTC）。

2. **Given** 已配置 Telegram 通知，**When** 发生交易执行（`trade` 事件），**Then** Telegram 消息包含：方向（买/卖）、数量、价格、预期盈亏比，格式为中文富文本。

3. **Given** 用户在 Telegram bot 中发送 `/status` 命令，**When** bot 接收到命令，**Then** bot 回复当前调度器状态（启用/禁用、下次运行时间、最近触发记录）。

4. **Given** 触发通知发送失败（网络超时），**When** 重试 3 次均失败，**Then** 系统记录警告日志但不阻塞主流程，不影响交易分析的正常执行。

---

### 用户故事 6 — 前端调度管理页 (Priority: P1)

**场景描述**：用户希望在 React 前端中有一个专门的"调度管理"页面，可以可视化查看、创建、编辑和删除调度规则，包括价格触发规则和定时规则，并查看触发历史记录。

**优先级说明**：目前前端仅有 Dashboard 中一个只读的 `SchedulerCard` 组件，无法管理调度规则。管理页面是使用价格触发功能的主要入口。

**独立验收测试**：打开调度管理页，创建一条价格触发规则，刷新页面后规则仍存在，触发后历史表格中出现记录。

**验收场景**：

1. **Given** 用户访问 `/scheduler` 页面，**When** 页面加载完成，**Then** 页面显示：当前所有调度规则列表（表格形式）、触发历史（最近 50 条）、快速创建按钮。

2. **Given** 用户点击"从模板创建"，**When** 选择模板（如"BTC 价格突破提醒"、"资金费率异常"），**Then** 弹出预填充的规则编辑表单，用户只需修改关键参数即可保存。

3. **Given** 用户使用 Cron 表达式编辑器创建定时规则，**When** 输入 `0 */4 * * *`，**Then** 编辑器实时显示中文说明"每 4 小时执行一次，下次执行：2026-04-17 20:00 UTC"，并在保存后加入规则列表。

4. **Given** 规则列表中有激活的规则，**When** 用户点击某规则的"禁用"开关，**Then** 规则状态立即更新为"已禁用"（不删除），调度器停止监听该规则。

5. **Given** 触发历史表格有记录，**When** 用户点击某条历史记录，**Then** 展开详情：触发原因、触发时的价格快照、触发后的分析结论（如果有）。

---

### 用户故事 7 — Agent 自我调度 (Priority: P2)

**场景描述**：用户希望 AI Agent 在分析过程中能够自主创建临时调度规则，例如当 Agent 判断"市场处于关键支撑位，需要在价格跌破 XX 时立即重新分析"时，可以自己注册一条临时价格触发规则。

**优先级说明**：这是最具创新性的功能，体现了 AI-first 理念，但依赖前几个阶段的基础设施，故为 P2。

**独立验收测试**：在 Agent 分析结论中提取 `schedule_follow_up` 字段，验证对应的临时触发规则被成功注册到调度器。

**验收场景**：

1. **Given** Agent 分析流程已完成，**When** 分析结论中包含 `schedule_follow_up` 字段（如 `{"type": "price_threshold", "pair": "BTC/USDT", "below": 79000, "ttl_hours": 24}`），**Then** 系统自动将该规则注册为一条 TTL 为 24 小时的临时价格触发规则，无需用户手动操作。

2. **Given** Agent 注册了临时触发规则，**When** TTL 到期且规则未被触发，**Then** 规则自动失效并从调度器移除，触发历史中记录"规则已过期"。

3. **Given** Agent 创建的临时规则被触发，**When** 重新触发的分析完成，**Then** 系统检查新分析结论是否再次包含 `schedule_follow_up`，若有则注册新规则，若没有则不继续（防止无限循环，最多递归 3 次）。

---

### 边界条件

- **同时触发防护**：同一交易对在同一时刻只能有一个分析周期运行（复用现有 `max_instances=1` 机制）
- **冷却期精度**：冷却期基于 Redis TTL（Redis 不可用时降级为内存计时），精度为秒级
- **WebSocket 断线重连**：断线后指数退避重连（1s, 2s, 4s, 最大 60s），重连期间不丢失规则配置
- **规则数量上限**：单用户场景下规则总数不超过 50 条（超出提示警告）
- **TTL 规则清理**：过期的临时规则由后台定期任务清理（每小时），不影响主循环性能
- **前端并发更新**：前端通过 React Query 轮询规则列表（间隔 5 秒），规则变更实时反映，无需手动刷新

## 需求规格

### 功能需求

#### 价格触发引擎

- **FR-001**：系统必须支持通过 Binance WebSocket 实时订阅一个或多个交易对的价格流（ticker stream），连接失败时自动重连。

- **FR-002**：系统必须支持以下四种触发类型：
  - **绝对阈值**（`price_threshold`）：价格高于或低于某个固定值
  - **百分比变化**（`pct_change`）：价格在指定时间窗口内涨跌超过 X%
  - **连续 K 线模式**（`candle_pattern`）：识别连续 N 根同方向 K 线
  - **资金费率异常**（`funding_rate`）：资金费率绝对值超过阈值

- **FR-003**：每条触发规则必须支持配置冷却期（`cooldown_minutes`，默认 30 分钟），冷却期内同一规则不重复触发。冷却期状态存储于 Redis（Redis 不可用时降级为进程内存）。

- **FR-004**：触发引擎必须支持优雅关闭——接收到 SIGTERM/SIGINT 时，等待当前进行中的分析周期完成后再关闭 WebSocket 连接，不丢失正在执行的任务。

- **FR-005**：每次触发事件必须记录到持久化存储（SQLite），包含：触发时间、触发规则 ID、触发类型、触发时的价格快照、冷却状态。

#### 通知系统扩展

- **FR-006**：系统必须在现有 `Notifier` 的基础上扩展对 Telegram Bot API 的原生支持，通过配置 `telegram_bot_token` 和 `telegram_chat_id` 启用。

- **FR-007**：Telegram 通知必须支持以下事件类型，并使用中文富文本格式（Markdown）：
  - `price_trigger`：价格触发器激活
  - `trade`：交易执行
  - `rejection`：风控拒绝
  - `circuit_breaker`：熔断器激活
  - `daily_summary`：每日汇总

- **FR-008**：Telegram bot 必须支持接收 `/status` 命令，回复当前系统状态快照（调度器状态、最近 3 次触发、Redis 可用性）。

- **FR-009**：通知发送失败时必须最多重试 3 次（指数退避），失败后记录警告日志，不阻塞主流程。

#### 调度规则持久化

- **FR-010**：调度规则（包括触发类型、参数、冷却期、启用状态、TTL）必须持久化到 SQLite 数据库，系统重启后规则自动恢复。

- **FR-011**：系统必须通过 FastAPI 提供调度规则的 CRUD API：
  - `GET /api/scheduler/rules` — 列出所有规则
  - `POST /api/scheduler/rules` — 创建规则
  - `PUT /api/scheduler/rules/{rule_id}` — 更新规则
  - `DELETE /api/scheduler/rules/{rule_id}` — 删除规则
  - `GET /api/scheduler/triggers` — 查询触发历史（支持分页）

- **FR-012**：规则 API 必须通过现有的 API Key 认证机制保护（复用 `security.py`）。

#### 前端调度管理页

- **FR-013**：前端必须新增 `/scheduler` 路由和对应页面，纳入 `App.tsx` 的路由体系，在侧边栏导航中显示。

- **FR-014**：调度管理页必须包含以下组件：
  - **规则列表表格**：显示规则名称、类型、交易对、状态（启用/禁用）、冷却状态、最近触发时间
  - **规则编辑表单**：支持所有四种触发类型的参数配置，含输入验证
  - **模板卡片选择器**：预置常用规则模板（不少于 4 个），点击后预填表单
  - **Cron 表达式编辑器**：输入 cron 表达式时实时显示中文自然语言说明及下次触发时间
  - **触发历史表格**：显示最近 50 条触发记录，支持展开查看详情

- **FR-015**：规则列表中每条规则必须支持单独的"启用/禁用"切换开关，状态变更通过 API 实时同步。

- **FR-016**：前端文本必须使用中文（与现有 i18n 架构一致，语言 key 存入 `locales/zh-CN/` 下的 scheduler 命名空间）。

#### Agent 自我调度

- **FR-017**：Agent 分析结论的 JSON 输出规格必须扩展支持可选字段 `schedule_follow_up`，用于声明后续触发规则。

- **FR-018**：`nodes/verdict.py` 或 `nodes/journal.py` 必须在分析流程结束后检查 `schedule_follow_up` 字段，若存在则自动调用规则创建 API 注册临时规则（TTL 上限 72 小时）。

- **FR-019**：Agent 自我调度的递归深度不超过 3 次（通过 `schedule_depth` 字段追踪），防止无限循环。

---

### 关键实体

- **ScheduleRule（调度规则）**：包含 `id`、`name`、`type`（触发类型枚举）、`pair`（交易对）、`parameters`（触发参数 JSON）、`cooldown_minutes`、`enabled`、`ttl_expires_at`（可选，Agent 规则专用）、`created_by`（`user` | `agent`）、`created_at`。

- **TriggerEvent（触发事件）**：包含 `id`、`rule_id`（外键）、`triggered_at`、`trigger_reason`（文本描述）、`price_snapshot`（触发时价格 JSON）、`analysis_commit_id`（可选，关联分析记录）、`schedule_depth`（Agent 规则递归深度）。

- **PriceTriggerEngine（价格触发引擎）**：负责维护 WebSocket 连接、匹配条件、执行冷却检查并派发触发事件的核心运行时组件。

- **TelegramNotifier（Telegram 通知器）**：作为 `Notifier` 的扩展，通过 Telegram Bot API 发送结构化中文消息，并处理 `/status` 命令的入站消息。

## 成功标准

### 可量化目标

- **SC-001**：价格突破绝对阈值后，系统在 **5 秒内**触发分析周期（WebSocket 延迟 <1s + 引擎处理 <4s）。

- **SC-002**：触发事件的**误触发率为零**——在冷却期内，同一规则不产生重复触发记录。

- **SC-003**：WebSocket 断线重连后，**100% 的已配置规则**自动恢复监听，无需重启系统。

- **SC-004**：Telegram 通知在触发事件发生后 **10 秒内**到达（包含网络传输时间）。

- **SC-005**：系统重启后，所有持久化规则**100% 恢复**到重启前的状态（启用/禁用、冷却期剩余时间通过 Redis TTL 恢复）。

- **SC-006**：前端调度管理页的初始加载时间（含规则列表）不超过 **1.5 秒**（P90，本地网络）。

- **SC-007**：所有新增后端模块的测试覆盖率不低于 **85%**，新增测试不依赖外部网络（WebSocket、Telegram API 通过 mock 测试）。

- **SC-008**：整体代码变更不引入新的 lint 错误（零 `ruff` 警告，禁止 `noqa` 注释）。

## 假设

- 本系统为**单用户自托管**场景，无多租户隔离需求；调度规则不需要用户维度的权限控制。
- Binance WebSocket API 的 ticker stream 稳定可用；若用户选择 OKX 作为交易所，价格触发器仍通过 Binance 公共 API 获取价格数据（与 `exchange_id` 设置解耦）。
- Redis 可选：冷却期状态优先存储于 Redis（与现有 `RedisStateManager` 模式一致），Redis 不可用时降级为进程内存（不跨进程持久化）。
- Telegram bot 为**单向通知**为主，`/status` 命令为唯一的入站命令（不支持通过 Telegram 创建或修改规则）。
- Agent 自我调度功能（P2）依赖 P0/P1 的 CRUD API 基础设施，必须在 P1 完成后才能开始实现。
- 前端使用与现有页面相同的技术栈（React 19、Vite 7、TypeScript 5.9 strict、React Query 5）和组件库（Radix UI / shadcn），不引入新的 UI 框架。
- 现有 `NotificationsConfig` 和 `Notifier` 类将被扩展（不替换），以保持向后兼容性。
